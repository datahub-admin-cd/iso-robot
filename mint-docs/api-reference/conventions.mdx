The ISO Robot API is a single versioned REST surface. This page covers the rules
that apply everywhere; the following pages document each endpoint group.

**Recommended integration:** [Automated Pipeline](/api-reference/pipeline) —
`POST /ingest/{clientOrgId}` plus `GET /pipeline/status/{clientOrgId}`.

## Base URL

Deployed API (VM):

```text
http://157.20.190.175:8000/api/v1
```

The root health check sits one level above the versioned prefix:

```text
http://157.20.190.175:8000/health
```

<Info>
Interactive OpenAPI docs on the same host: `http://157.20.190.175:8000/docs`
</Info>

## Authentication

All endpoints require a bearer token **except** the public ones below.

```http
Authorization: Bearer <access_token>
```

| Route | Auth |
| --- | --- |
| `POST /auth/login`, `POST /auth/register`, `GET /health` | **None** |
| Everything else | Bearer token required |

Obtain the token from [`POST /auth/login`](/api-reference/authentication). The
login response also returns the `client_org_id` that scopes most resources.

## Response envelope

Most write/detail endpoints wrap their payload in a standard envelope:

```json
{
  "status": "success",
  "message": "…",
  "data": { }
}
```

List endpoints frequently return a **bare JSON array** (for example `/controls`,
`/issues`, `/candidate-risks`). Each endpoint page notes which shape it uses.

## Error format

Every error uses the same body:

```json
{ "detail": "Human-readable message", "code": "ERROR_CODE" }
```

| HTTP | code | Meaning |
| --- | --- | --- |
| 401 | `UNAUTHORIZED` | Missing or invalid bearer header |
| 401 | `SESSION_INVALID` | Expired or malformed JWT |
| 401 | `AUTH_FAILED` | Wrong username / password |
| 403 | `FORBIDDEN` | Org mismatch or insufficient role |
| 404 | `not_found` | Resource missing |
| 404 | `CLIENT_ORG_NOT_FOUND` | Unknown organisation id |
| 409 | `DUPLICATE_RECORD` | Slug / email already taken |
| 422 | `validation_error` | Invalid request body |

## Background jobs & pipeline runs

Long-running work is asynchronous. Two polling models exist:

| Model | Start | Poll | Use when |
| --- | --- | --- | --- |
| **Automated pipeline** | `POST /ingest/{clientOrgId}` | `GET /pipeline/status/{clientOrgId}` | Production ingest, webhooks |
| **Legacy jobs** | Per-stage POST (e.g. `/control-documents/extract/{orgId}`) | `GET /jobs/{jobId}` | Debugging, single-stage re-runs |

The automated pipeline is documented in [Pipeline API](/api-reference/pipeline)
and [Background Jobs](/process/background-jobs). Legacy job endpoints are on
[Jobs](/api-reference/jobs).

## Collection variables

The API verification suite uses these variables; they map directly to values you
capture during a run:

| Variable | Source |
| --- | --- |
| `token` | `access_token` from login |
| `clientOrgId` | `client_org_id` from login |
| `pipelineRunId` | `pipeline_run_id` from ingest (`202` response) |
| `documentId` | a control document id |
| `issueId` | an issue id |
| `jobId` | a job id returned by any legacy start endpoint |
